Introduction

FAQ

  1. Qu'est-ce que c'est?

    Il s'agit d'un module externe qui permet aux administrateurs REDCap de créer et de gérer des groupes d'accès de sécurité dans un système REDCap.

  2. Que sont les groupes d’accès de sécurité ?

    Les groupes d'accès de sécurité (SAG) sont utilisés pour restreindre les droits d'utilisateur qu'un utilisateur REDCap peut obtenir dans un projet. Les SAG ne définissent pas les droits qu'un utilisateur aura dans un projet donné ; ils définissent plutôt l'ensemble des droits autorisés dont l'utilisateur peut se voir accorder. Les SAG sont définis au niveau du système et sont utilisés dans tout projet pour lequel ce module est activé.

  3. Quand est-ce que cela serait utile ?

    Par exemple, si votre institution exige une documentation sur la formation des utilisateurs pour exporter des données, vous pouvez créer un SAG qui n'autorise pas l'octroi du droit d'utilisateur « Outil d'exportation de données » aux utilisateurs qui lui sont attribués. Une fois qu'un utilisateur a terminé sa formation, il peut être déplacé vers un SAG qui permet d'accorder le droit d'utilisateur « Outil d'exportation de données ».

    En d’autres termes, vous souhaiterez peut-être proposer différents niveaux de formation à vos utilisateurs et accorder uniquement des droits d’utilisateur spécifiques au niveau du projet aux utilisateurs qui ont suivi la formation appropriée. Bien que ce module n'administre ni ne suit la formation, il gère tout le reste.

  4. N'est-ce pas la responsabilité de l'équipe d'étude de gérer les droits des utilisateurs ?

    Oui et non. En tant qu'administrateurs système, nous avons la responsabilité de garantir que les utilisateurs de notre système sont correctement formés pour le travail qu'ils effectueront. De nombreux utilisateurs ne souhaitent pas suivre une formation d'une heure s'ils se contentent de saisir des données dans un instrument de collecte de données. Vous pouvez offrir à ces utilisateurs une formation plus courte et ciblée et les affecter à un SAG qui accorde des droits d'utilisateur minimaux.

Comment utilisez-vous le module ?

Les administrateurs REDCap créent d'abord des SAG à l'aide d'un menu dans le centre de contrôle. Une fois un SAG créé, les utilisateurs peuvent lui être attribués dans un autre onglet du même menu du centre de contrôle. Lorsqu'un utilisateur est affecté à un SAG, il ne pourra se voir accorder que les droits d'utilisateur autorisés par le SAG.

🚩
 Le module doit être activé dans un projet pour pouvoir restreindre les droits d'utilisateur dans ce projet. Il est recommandé que le module soit activé dans tous les projets.

Lorsque le module est activé dans un projet, il vérifiera quels droits sont accordés à un utilisateur en fonction de son SAG avant d'accorder des droits d'utilisateur à cet utilisateur. Le module intercepte toutes les tentatives de définition des droits des utilisateurs dans un projet, y compris

  • En utilisant les méthodes habituelles sur la page du projet Droits des utilisateurs

  • Utilisation de la fonctionnalité Télécharger des utilisateurs, des rôles et des affectations sur la page du projet Droits des utilisateurs

  • Utilisation de l'API REDCap

Toutes les tentatives de définition des droits de l'utilisateur seront bloquées si l'utilisateur n'est pas autorisé à se voir accorder les droits conformément à son SAG. Les tentatives de contournement du module sont détectées et enregistrées.

💡

Le module ignore les utilisateurs actuellement expirés dans le projet. Cela permet à l'utilisateur de rester dans le projet mais l'empêche d'utiliser ses droits d'utilisateur non conformes jusqu'à ce que ces droits non conformes soient supprimés. L'utilisateur peut alors être réexpiré.

Le module fournit un moyen pratique de faire expirer les utilisateurs d'un projet dans l' onglet Statut du projet .

Onglet Rapports

Figure 11. Onglet Rapports

Cet onglet offre un moyen simple de voir tous les utilisateurs du système qui disposent actuellement de droits d'utilisateur qui ne sont pas conformes à leur SAG actuel. Cela peut se produire lorsque le module est activé pour la première fois dans un projet ou lorsqu'un utilisateur est affecté à un nouveau SAG.

Les options de rapport sont les suivantes :

Titre du rapport Description

Utilisateurs avec des droits non conformes (non expirés)

Ce rapport répertorie tous les utilisateurs affectés à un SAG qui ne permet pas à l'utilisateur de bénéficier de tous les droits dont il dispose actuellement dans un projet. Ce rapport inclut uniquement les utilisateurs s'ils n'ont pas actuellement expiré dans le(s) projet(s).

Utilisateurs avec des droits non conformes (tous)

Ce rapport répertorie tous les utilisateurs affectés à un SAG qui ne permet pas à l'utilisateur de bénéficier de tous les droits dont il dispose actuellement dans un projet. Ce rapport inclut tous les utilisateurs, qu'ils aient ou non expiré dans le(s) projet(s).

Projets avec des droits non conformes (non expirés)

Ce rapport répertorie tous les projets qui ont au moins un utilisateur affecté à un SAG qui ne permet pas à l'utilisateur de bénéficier de tous les droits dont il dispose actuellement dans le projet. Ce rapport inclut uniquement les utilisateurs disposant d'un compte utilisateur non expiré.

Projets avec des droits non conformes (tous)

Ce rapport répertorie tous les projets qui ont au moins un utilisateur affecté à un SAG qui ne permet pas à l'utilisateur de bénéficier de tous les droits dont il dispose actuellement dans le projet. Ce rapport inclut tous les utilisateurs, que leur compte utilisateur soit expiré ou non.

Utilisateurs et projets avec des droits non conformes (non expirés)

Ce rapport répertorie toutes les combinaisons d'utilisateur et de projet dans lesquelles l'utilisateur est affecté à un SAG qui ne lui permet pas d'obtenir tous les droits dont il dispose actuellement dans le projet. Ce rapport inclut uniquement les utilisateurs qui n'ont pas actuellement expiré dans le projet.

Utilisateurs et projets avec des droits non conformes (tous)

Ce rapport répertorie toutes les combinaisons d'utilisateur et de projet dans lesquelles l'utilisateur est affecté à un SAG qui ne lui permet pas d'obtenir tous les droits dont il dispose actuellement dans le projet. Ce rapport inclut tous les utilisateurs, qu'ils aient ou non expiré dans le projet.

Figure 12. Exemple de rapport
💡
 Vous pouvez filtrer en fonction du statut du projet en incluant "project_status=" puis le statut sur lequel vous souhaitez filtrer. Par exemple, pour inclure uniquement les projets en production, vous utiliserez project_status=Production . Pour inclure uniquement les projets terminés, vous devez utiliser project_status=Completed . Pour inclure à la fois les projets de production et de développement, vous devez utiliser project_status=Production | project_status=Development . Voir la section Recherche pour plus d'informations.

Page du projet

Onglet Statut du projet

Le module ajoute une page qui affiche le statut de tous les utilisateurs du projet. Le statut de chaque utilisateur est déterminé par le SAG de l'utilisateur et les droits dont il dispose dans le projet. La couleur de la ligne indique si l'utilisateur est :

  • Vert - conforme à leur SAG

  • Rouge - non conforme à leur SAG

  • Gris - expiré dans le projet

Vous pouvez également consulter la colonne Droits non conformes pour voir les droits dont dispose l'utilisateur qui ne sont pas autorisés par son SAG.

💡
 Si vous souhaitez voir uniquement les utilisateurs qui disposent de droits particuliers de manière inappropriée, vous pouvez utiliser le champ de recherche. Par exemple, si vous êtes uniquement intéressé par les droits d'utilisateur et/ou les droits de conception et de configuration de projet , tapez "user rights" | "project design" dans la zone de recherche. Voir la section Recherche pour plus d'informations.
Figure 14. Onglet Statut du projet
💡
 Si des utilisateurs ne respectent pas leur SAG, vous pouvez utiliser l'un des boutons d'action pour envoyer un e-mail à l'utilisateur, aux titulaires de droits de l'utilisateur, ou aux deux. Vous pouvez également faire expirer l'utilisateur du projet. Une alerte peut éventuellement être envoyée à l'utilisateur et/ou aux ayants droit de l'utilisateur lorsque l'utilisateur est expiré.

Enregistrement

L'un des avantages de l'utilisation de ce module est la journalisation améliorée qu'il fournit. Le module crée des journaux détaillés dans les propres journaux du projet pour toutes les modifications apportées aux droits des utilisateurs, y compris

  • Lorsqu'un utilisateur est ajouté à un projet avec des droits personnalisés

  • Lorsqu'un utilisateur est ajouté à un projet dans un rôle d'utilisateur existant

  • Lorsque les droits d'un utilisateur sont modifiés

  • Lorsque la définition des droits d'un rôle est modifiée

  • Lorsqu'un utilisateur est ajouté/supprimé d'un rôle d'utilisateur

  • Lorsque les utilisateurs sont importés dans un projet via CSV

  • Lorsque les droits d'un utilisateur sont modifiés via l'importation CSV

  • Lorsque les rôles sont importés dans un projet via CSV

  • Lorsqu'un utilisateur est attribué à un rôle via l'importation CSV

  • Lorsqu'un utilisateur est ajouté à un projet via l'API

  • Lorsque les droits d'un utilisateur sont modifiés via l'API

  • Lorsque les rôles d'utilisateur sont importés/modifiés via l'API

  • Lorsqu'un utilisateur est affecté à un rôle via l'API

Traduction (i18n / l10n)

Informations générales

Le module peut être traduit dans des langues autres que l'anglais via la fonction de sélection de langue de l'External Module Framework. Pour définir la langue du module à l'échelle du système, accédez à Centre de contrôle > Modules externes > Gérer les modules externes et cliquez sur le bouton « Configurer » du module. Sélectionnez ensuite la langue que vous souhaitez utiliser dans la liste déroulante « Langue » et cliquez sur « Enregistrer ».

Vous pouvez remplacer le langage à l'échelle du système au niveau du projet en visitant le gestionnaire de module de projet et en cliquant sur le bouton « Configurer » du module. Sélectionnez ensuite la langue que vous souhaitez utiliser dans la liste déroulante « Langue » et cliquez sur « Enregistrer ».

Limites

Langues de droite à gauche

Actuellement, la prise en charge des langues RTL est limitée. Le module affichera correctement les langues RTL, mais la structure/formatage des éléments de l'interface utilisateur sera toujours LTR. Il existe des options pour une prise en charge plus complète de RTL, mais cela ne sera qu'une faible priorité à moins que nous n'entendions des groupes qui ont besoin de cette fonctionnalité.

Traduction du module

Les traductions fournies avec le module ont été créées à l'aide d'un logiciel de traduction automatique et peuvent ne pas être exactes. Si vous souhaitez corriger une traduction ou contribuer à une nouvelle traduction, veuillez suivre les instructions ci-dessous.

Créer une nouvelle traduction

Si vous souhaitez traduire le module dans une nouvelle langue, commencez par créer la branche main du référentiel Github du module. Ensuite, suivez ces étapes :

  1. Recherchez le répertoire lang dans le code source du module.

  2. Copiez le fichier English.ini et remplacez le nom de la copie par la langue dans laquelle vous souhaitez traduire. Nommez le fichier avec le nom anglais de la langue (en majuscule) suivi de l'endonyme de la langue (en utilisant les glyphes de cette langue) entre parenthèses. Par exemple, si vous souhaitez traduire le module en japonais, vous nommerez le fichier Japanese (日本語).ini .

  3. Ouvrez le fichier que vous venez de créer dans un éditeur de texte et traduisez le texte à droite du signe égal pour chaque ligne. Par exemple, si vous souhaitez traduire le texte Introduction en japonais, vous devez remplacer la ligne status_ui_3 = "Introduction" par status_ui_3 = "序章" .

  4. Répétez le processus pour chaque ligne.

  5. Enregistrez le fichier et téléchargez-le dans le dossier "lang" du code source du module.

  6. Soumettez une pull request avec vos modifications à la branche main du référentiel Github .

Corriger une traduction existante

Si vous souhaitez corriger une traduction existante, vous pouvez le faire en suivant ces étapes :

  1. Recherchez le répertoire lang dans le code source du module.

  2. Ouvrez le fichier de la langue que vous souhaitez corriger dans un éditeur de texte.

  3. Corrigez le texte à droite du signe égal pour chaque ligne que vous souhaitez modifier.

  4. Enregistrez le fichier et téléchargez-le dans le dossier "lang" du code source du module.

  5. Soumettez une pull request avec vos modifications à la branche main du référentiel Github .

Divers

Recherche

Général

De nombreux tableaux du module disposent d'un champ de recherche qui peut être utilisé pour filtrer le tableau. La zone de recherche recherchera toutes les colonnes du tableau.

Par exemple, si vous souhaitez rechercher tous les utilisateurs actuellement affectés au SAG dont l'étiquette contient le mot 'Nothing' , vous pouvez saisir 'Nothing' dans la zone de recherche et le tableau sera filtré pour afficher uniquement les lignes qui ont le texte 'Nothing' dans n'importe quelle colonne.

Figure 26. Exemple de recherche

Recherches "ET"

Par défaut, le champ de recherche sera une recherche « ET », ce qui signifie qu'il divisera votre terme de recherche en mots distincts et affichera uniquement les résultats qui correspondent à tous ces mots de recherche. Par exemple, si vous tapez 'joe admin' dans la zone de recherche, le tableau sera filtré pour afficher uniquement les lignes contenant à la fois 'joe' ET 'admin' dans n'importe quelle colonne. Il n'affichera pas les lignes contenant 'joe' ou 'admin' , mais il affichera les lignes contenant par exemple 'admin' dans la colonne 1 et 'joe' dans la colonne 2.

Figure 27. Exemple ET

Recherches « OU »

Si vous souhaitez afficher toutes les lignes qui correspondent à une valeur OU correspondent à une autre valeur, vous devez utiliser une recherche « OU ».

Vous pouvez utiliser le "|" opérateur pour rechercher plusieurs termes comme celui-ci de manière soit/ou. Par exemple, si vous souhaitez rechercher tous les utilisateurs portant le nom d'utilisateur "alice" ou "bob" , vous pouvez saisir "alice | bob" dans la zone de recherche et le tableau sera filtré pour afficher uniquement les utilisateurs portant "alice" ou "bob" dans n'importe quelle colonne.

💡
 y compris le '|' Le symbole a pour effet secondaire de faire de la recherche une recherche d'expression régulière (voir ci-dessous ).
Figure 28. Exemple de OU

Recherches d’expressions exactes

Si vous souhaitez filtrer en fonction d'une phrase, vous pouvez mettre votre phrase entre guillemets doubles. Par exemple, si vous souhaitez rechercher toutes les lignes contenant l'expression exacte 'joe admin' vous pouvez taper '"joe admin"' dans la zone de recherche et le tableau sera filtré pour afficher uniquement les lignes avec 'joe admin' dans n'importe quelle colonne.

Figure 29. Exemple de phrase exacte

Recherches d'expressions régulières

En incluant un '|' caractère n'importe où dans votre terme de recherche, votre recherche se transforme en une recherche d'expression régulière . Cela vous permet de composer des recherches complexes qui ne sont pas possibles avec la recherche par défaut.

Figure 30. Exemple d'expression régulière - affichant toutes les lignes où un "d" est suivi d'un nombre quelconque de lettres, puis d'un "n" - comme dans "admin" et "dan"
